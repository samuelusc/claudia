1. Executive Summary

The Claudia Checkpoint System is a sophisticated version control and session management system designed specifically for Claude Code interactions. Built as a core component of the Claudia desktop application, it provides developers with powerful tools for managing, tracking, and restoring different states of their AI-assisted coding sessions.

рҹ”„ Session Versioning

Create checkpoints at any point in your coding session, enabling instant rollbacks and state exploration.

рҹҢі Branching Timeline

Fork sessions from any checkpoint to explore alternative approaches without losing previous work.

рҹ“Ғ Smart File Tracking

Automatically detect and snapshot file changes with content-addressable storage for efficiency.

рҹҡҖ Performance Optimized

zstd compression and deduplication ensure minimal storage overhead and fast operations.

Key Benefits:

  • Risk-free experimentation: Try different approaches without fear of losing working code
  • Efficient collaboration: Share specific states and branches with team members
  • Debugging assistance: Quickly identify when and where issues were introduced
  • Learning enhancement: Review the evolution of solutions and decision points

The system is architected with a Rust backend for performance and safety, integrated with a React + TypeScript frontend via Tauri 2, providing a seamless native desktop experience.

Architecture Principles

  • Separation of Concerns: Clear boundaries between UI, business logic, and storage
  • Thread Safety: Arc<RwLock> patterns for safe concurrent access
  • Content Addressability: SHA-256 based file storage for deduplication
  • Immutable Timelines: Checkpoint data is never modified after creation
  • Lazy Loading: Components and data loaded on-demand for performance

Communication Flow

  1. User Action: Frontend component triggers checkpoint operation
  2. IPC Call: Tauri command invoked with session parameters
  3. Manager Resolution: CheckpointState locates or creates appropriate manager
  4. Business Logic: CheckpointManager executes operation logic
  5. Storage Operation: CheckpointStorage handles persistence
  6. Response: Results propagated back through the chain to UI

3. Core Components

The checkpoint system consists of four primary components that work together to provide seamless version control for Claude Code sessions.

3.1 CheckpointManager

The CheckpointManager serves as the central coordinator for all checkpoint operations within a specific session. It maintains session state, tracks file modifications, and orchestrates the creation and restoration of checkpoints.

pub struct CheckpointManager {
    // Session identification
    project_id: String,
    session_id: String,
    project_path: PathBuf,
    
    // State management
    file_tracker: Arc<RwLock<FileTracker>>,
    storage: Arc<CheckpointStorage>,
    timeline: Arc<RwLock<SessionTimeline>>,
    current_messages: Arc<RwLock<Vec<String>>>,
}

Key Responsibilities

  • Message Tracking: Monitors Claude Code session messages in JSONL format
  • File Change Detection: Automatically detects file modifications from tool usage
  • Auto-Checkpoint Triggers: Implements various checkpoint strategies
  • Timeline Management: Maintains the session's timeline tree structure

Auto-Checkpoint Strategies

Manual

Checkpoints created only on explicit user request

PerPrompt

Automatic checkpoint after each user prompt

PerToolUse

Checkpoint triggered by any tool execution

Smart (Default)

Intelligent checkpointing based on file modifications and operation types

3.2 CheckpointState

The CheckpointState component provides a stateful cache of CheckpointManager instances, ensuring efficient resource utilization and preventing unnecessary recreations of manager objects.

pub struct CheckpointState {
    // Thread-safe manager cache
    managers: Arc<RwLock<HashMap<String, Arc<CheckpointManager>>>>,
    claude_dir: Arc<RwLock<Option<PathBuf>>>,
}

Benefits of Manager Caching

  • Performance: Avoids expensive re-initialization of session state
  • Memory Efficiency: Shared references prevent duplication
  • State Consistency: Maintains file tracking state across operations
  • Thread Safety: Safe concurrent access from multiple Tauri commands

3.3 CheckpointStorage

The CheckpointStorage component handles all persistent storage operations, implementing an efficient content-addressable storage system with compression and deduplication.

pub struct CheckpointStorage {
    claude_dir: PathBuf,
    compression_level: i32, // zstd compression level (default: 3)
}

Storage Optimizations

  • Content-Addressable Storage: Files stored by SHA-256 hash, enabling automatic deduplication
  • zstd Compression: Message and metadata compression with configurable levels
  • Incremental Snapshots: Only modified files are stored per checkpoint
  • Integrity Verification: Hash-based validation prevents data corruption

4. Key Data Structures

The checkpoint system defines several crucial data structures that represent different aspects of session state and metadata.

Checkpoint

The core data structure representing a single checkpoint in the timeline.

pub struct Checkpoint {
    // Unique identification
    pub id: String,                    // UUID v4
    pub session_id: String,            // Parent session
    pub project_id: String,            // Project context
    
    // Timeline position
    pub message_index: usize,           // Last message included
    pub parent_checkpoint_id: Option<String>, // For branching
    
    // Temporal data
    pub timestamp: DateTime<Utc>,        // Creation time
    pub description: Option<String>,     // User description
    
    // Associated metadata
    pub metadata: CheckpointMetadata,    // Detailed information
}

CheckpointMetadata

Rich metadata associated with each checkpoint for analytics and navigation.

pub struct CheckpointMetadata {
    // Usage tracking
    pub total_tokens: u64,               // Cumulative token usage
    pub model_used: String,              // Claude model version
    
    // Context information
    pub user_prompt: String,             // Triggering prompt
    pub file_changes: usize,             // Number of files modified
    pub snapshot_size: u64,             // Total storage size
}

FileSnapshot

Represents the state of a single file at a specific checkpoint.

pub struct FileSnapshot {
    // Association
    pub checkpoint_id: String,           // Parent checkpoint
    pub file_path: PathBuf,             // Relative to project
    
    // Content data
    pub content: String,               // File content (compressed)
    pub hash: String,                  // SHA-256 hash
    pub size: u64,                     // Original file size
    
    // File system attributes
    pub permissions: Option<u32>,       // Unix permissions
    pub is_deleted: bool,               // Deletion marker
}

TimelineNode

Tree structure node for representing checkpoint relationships.

pub struct TimelineNode {
    // Core checkpoint data
    pub checkpoint: Checkpoint,
    
    // Tree relationships
    pub children: Vec<TimelineNode>,       // Child branches
    
    // Associated resources
    pub file_snapshot_ids: Vec<String>,    // File references
}

8.3 Storage Mechanisms and Optimization

Content-Addressable Storage Implementation

// SHA-256 based file storage
fn save_file_snapshot(
    &self, 
    paths: &CheckpointPaths, 
    snapshot: &FileSnapshot
) -> Result<()> {
    // Calculate content hash
    let mut hasher = Sha256::new();
    hasher.update(snapshot.content.as_bytes());
    let hash = format!("{:x}", hasher.finalize());
    
    // Store in content-addressable pool
    let content_path = paths.files_dir
        .join("content_pool")
        .join(&hash);
    
    // Only write if file doesn't exist (deduplication)
    if !content_path.exists() {
        let compressed = encode_all(
            snapshot.content.as_bytes(), 
            self.compression_level
        )?;
        fs::write(&content_path, compressed)?;
    }
    
    // Create reference file
    let reference = FileReference {
        checkpoint_id: snapshot.checkpoint_id.clone(),
        original_path: snapshot.file_path.clone(),
        content_hash: hash,
        size: snapshot.size,
        permissions: snapshot.permissions,
    };
    
    self.save_file_reference(paths, &reference)?;
    Ok(())
}

use tokio::task::JoinSet;

async fn save_file_snapshots_parallel(
    snapshots: Vec<FileSnapshot>
) -> Result<usize> {
    let mut join_set = JoinSet::new();
    
    for snapshot in snapshots {
        join_set.spawn(async move {
            self.save_file_snapshot(&paths, &snapshot)

    Ok(success_count)
